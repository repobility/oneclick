# Architecture

Server-rendered HTML + vanilla-JS crypto path. The Go server holds only
ciphertext; the AES-GCM key never leaves the browser.

```
       ┌──────────────────────────────────────┐
       │           Compose page               │
       │           (templates/create.html +   │
       │            static/compose.js)        │
       │                                      │
       │  1. user types a URL                 │
       │  2. crypto.getRandomValues(32) → key │
       │  3. crypto.getRandomValues(12) → IV  │
       │  4. crypto.subtle.encrypt(           │
       │       AES-GCM, key, iv, url)         │
       │  5. POST { ciphertext, iv, ttl,      │
       │           max_clicks } to            │
       │     /api/links                       │
       └──────────────────────────────────────┘
                          │
                          ▼  HTTPS
   ┌────────────────────────────────────────────────────────────────────┐
   │                  Go server (main.go + store.go)                     │
   │                                                                    │
   │  POST /api/links              → validate, mint 22-char id, insert  │
   │  GET  /api/links/{id}/meta    → non-consuming peek                 │
   │  GET  /api/links/{id}         → atomic consume (returns ciphertext │
   │                                  + iv, decrements or deletes)      │
   │  GET  /l/{id}                 → viewer SPA shell (html/template)   │
   │  GET  /                       → compose SPA shell                  │
   │  GET  /healthz                → liveness probe                     │
   │  GET  /robots.txt, /sitemap.xml, /humans.txt, /llms.txt,           │
   │      /.well-known/security.txt → hygiene files                     │
   │  GET  /static/*               → js + css                            │
   │                                                                    │
   │  Storage:  SQLite (modernc.org/sqlite — pure Go, no CGo).          │
   │            One row per link with ciphertext, iv, expires_at,       │
   │            clicks_remaining, created_at.                            │
   │                                                                    │
   │  Periodic sweep: every 60s, drop rows past expires_at.             │
   │                                                                    │
   │  The server holds NO keys. It cannot decrypt anything.             │
   └────────────────────────────────────────────────────────────────────┘
                          │
                          ▼  HTTPS
       ┌──────────────────────────────────────┐
       │           Viewer page                │
       │           (templates/view.html +     │
       │            static/view.js)           │
       │                                      │
       │  6. parse /l/<id>#k=<key> from URL   │
       │  7. peek /api/links/:id/meta         │
       │     → show "this consumes a click"   │
       │  8. user confirms                    │
       │  9. GET /api/links/:id consumes      │
       │     and returns ct + iv              │
       │ 10. crypto.subtle.decrypt(           │
       │       AES-GCM, key, iv, ct)          │
       │ 11. 600ms anti-phishing preview      │
       │ 12. location.replace(plaintext)      │
       └──────────────────────────────────────┘
```

## Layers

### 1. Crypto (`static/crypto.js`)

Pure browser code; uses `crypto.subtle` + `crypto.getRandomValues`:

- `ocEncrypt(plaintext)` — fresh 32-byte key + 12-byte IV, returns
  `{ ciphertext, iv, key }` (key is raw bytes, the others are base64).
- `ocDecrypt(ciphertext, iv, key)` — returns plaintext string or `null`
  on every failure mode.
- Base64 + base64url helpers for fragment embedding.

### 2. Server (`main.go` + `store.go`)

`main.go` wires the chi router, registers handlers, sets up the
`securityHeaders` middleware, starts the periodic sweep goroutine,
and runs the HTTP server.

`store.go` is a thin SQLite wrapper. `Consume` is the only multi-step
operation — it runs inside a transaction so decrement-or-delete is
atomic with the read.

The 16-byte id is generated by `crypto/rand` (Go stdlib CSPRNG) and
base64url-encoded to a 22-char string. Collisions are vanishingly
rare; the insert path retries up to 5 times on the unique-constraint
violation.

### 3. Compose + viewer (`templates/` + `static/{compose,view}.js`)

Server-rendered HTML shells via Go's `html/template`. The viewer
template embeds the URL id into a `<script type="application/json">`
block (which CSP treats as data, not script) so `view.js` reads it
on load without parsing `location.pathname` again.

### 4. Tests (`*_test.go`)

Three suites, all under Go's stdlib testing package:

- `store_test.go` — exercises the SQLite layer in isolation against
  per-test temp databases.
- `handlers_test.go` — drives the chi router with `httptest`, no
  real network involved.
- `auth_test.go` — six AUTH-* cases that prove the capability-URL
  matrix in `.repobility/access.yml`.

## Threat model summary

See [SECURITY.md](SECURITY.md). Headline:

| Adversary                         | Defended?                               |
| --------------------------------- | --------------------------------------- |
| Honest-but-curious operator       | ✅ confidentiality and integrity        |
| Passive network observer (w/ TLS) | ✅                                      |
| Active MITM (w/ TLS)              | ✅ — AES-GCM tag catches tampering      |
| URL leakage (browser history)     | ⚠️ one-time-click is the mitigation    |
| Referer leakage from viewer page  | ✅ `no-referrer` meta + header          |
| Phishing via the redirect         | ⚠️ explicit confirm + 600ms preview    |
| Compromised browser / XSS         | ⚠️ best-effort via strict CSP          |
| Compromised server delivering JS  | ⚠️ trust-the-server-once problem       |
