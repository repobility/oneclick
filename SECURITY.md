# Security model

This is the long-form companion to the security notes in [README.md](README.md).
It states what OneClick defends against, what it intentionally doesn't,
and the cryptographic reasoning behind both.

---

## Cryptographic primitive

OneClick uses the browser's **Web Crypto API** with AES-GCM:

- 256-bit symmetric key from `crypto.getRandomValues(new Uint8Array(32))`.
- 96-bit (12-byte) IV from `crypto.getRandomValues(new Uint8Array(12))`.
- AES-GCM authenticated encryption; the 128-bit tag is appended to the
  ciphertext in the standard Web Crypto layout.

This is a NIST-standardised, hardware-accelerated AEAD on every modern
platform. The Web Crypto API is exposed under `window.crypto.subtle` in
all evergreen browsers.

### Key derivation

None. The 32-byte key is generated by `getRandomValues`, exported as
raw bytes, and embedded in the URL fragment.

### Nonce reuse

Each upload generates a fresh 12-byte IV. Each key is one-shot — used
exactly once per encryption then garbage-collected, so (key, IV) reuse
is impossible by construction.

## Threat model

### 1. Honest-but-curious operator

**Defended.** The server stores only `{ id, ciphertext, iv, expires_at,
clicks_remaining }`. The key is in the URL fragment — browsers never
send fragments in HTTP. The server cannot decrypt, even with full disk
and log access.

Operator can see: that a link existed, its size (a fixed ~constant for
a URL-sized plaintext), the TTL, the click cap, and roughly when it
was created and visited. Operator cannot see: the destination URL or
the key.

### 2. Passive observer

**Defended, conditional on TLS.** OneClick must be deployed behind
HTTPS. With TLS, the observer sees only encrypted bytes plus IP-level
metadata. Without TLS, AES-GCM still holds for confidentiality, but
IPs and timing are exposed.

### 3. Active MITM

**Defended with TLS pinning.** A MITM without the URL-fragment key
cannot decrypt. Tampering with the ciphertext or the IV causes
`crypto.subtle.decrypt` to throw, which the viewer surfaces as a
generic "Could not decrypt" — never a falsified destination URL.

### 4. URL leakage (the dominant realistic risk)

**Partially defended.** A capability URL is only as safe as its
distribution channel:

- **Browser history**: the full URL with fragment is stored locally.
  Once the link is consumed, the server copy is gone — the URL in
  history becomes worthless.
- **Referer header**: the viewer page sets `<meta name="referrer"
  content="no-referrer">` and the server sets `Referrer-Policy:
  no-referrer`, so the destination URL doesn't leak via Referer when
  the viewer redirects.
- **Server logs**: HTTP servers cannot log fragments by construction.
- **Chat/email middlemen**: anyone who can read the channel can open
  the link. Use a private channel.

### 5. Compromised browser / XSS

**Best-effort defence.** Mitigations:

- Strict CSP applied by the `securityHeaders` middleware:
  `default-src 'self'; script-src 'self'; connect-src 'self';
  frame-ancestors 'none'`. No third-party origins.
- No `innerHTML` with user content — the templates render via
  `html/template`, which escapes by default.

### 6. Phishing via decrypted redirect

**Mitigated.** The viewer requires an explicit "Decrypt &amp; open"
click before the redirect, and shows a 600ms preview of the
destination URL so the user can abort.

### 7. Compromised server delivering a backdoored bundle

**Not defended in code.** The standard trust-the-server-once problem
of any web-delivered E2E tool.

## Server-side validation

The server never trusts client input. Every value is checked before
state mutation:

| Field           | Rule                                                        |
| --------------- | ----------------------------------------------------------- |
| `ciphertext`    | Base64, ≥ 24 chars (16-byte AES-GCM tag), ≤ 16 KiB.         |
| `iv`            | Exactly 16 base64 chars → 12 bytes after decoding.          |
| `ttl_seconds`   | Clamped to [60, 604800] (1 minute to 7 days).               |
| `max_clicks`    | Clamped to [1, 100].                                        |
| Request body    | `http.MaxBytesReader` capped at 64 KiB.                     |
| `id` (path)     | Exactly 20–32 chars, urlsafe-base64 alphabet.               |

The atomic click-consume + record-delete logic lives entirely inside
the `Store.Consume` transaction.

## Intentional non-goals

- **No DELETE / edit / list endpoints.** Once posted, a link cannot
  be retracted, modified, or enumerated.
- **No click receipts.** The creator cannot learn whether their link
  was opened.
- **No passphrase protection.** Could be added with WebCrypto's
  `deriveKey` over a passphrase; not in v1.
- **No multi-recipient feature.** The model is one URL = one capability.
- **No federation.** Each OneClick instance is its own namespace.

## Reporting a vulnerability

Please **do not** open a public issue. Email the maintainer first so
a fix can be coordinated before disclosure. See
`/.well-known/security.txt` for the contact and policy URL.
